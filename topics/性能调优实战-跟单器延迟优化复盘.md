# 性能调优实战：跟单器延迟优化复盘

> 日期：2026-07-19
> 标签：`性能调优` `数据库优化` `系统调试` `工程权衡` `幂等性`
> 类型：主题笔记（实战复盘）

---

## 背景

这次跟单器（copy trading）延迟优化是一个很典型的**性能调优实战**。它涉及的不是单一的算法题，而是一个真实混合系统：需要跨语言、跨进程地定位瓶颈、分层判断、小步快跑地优化。以下是这个过程对计算机学习的几个层面的帮助与沉淀。

---

## 1. 性能分析的核心方法：先测量，再优化

很多人调性能一上来就改代码，但这里的第一步是**加日志、看数据**：

- 先看 `snapshot synced in X ms` 总延迟
- 再拆出 `order_send completed in X ms`
- 最后把中间阶段逐个打出来

**核心原则：瓶颈永远由数据决定，不是由感觉决定。** 你学到的不是"怎么优化 SQLite"，而是"如何找到真正的瓶颈"。

---

## 2. 延迟分层思维

这个项目把延迟自然地分成了几层：

| 层级 | 代表耗时 | 可控性 |
|---|---|---|
| MT4 EA 检测 | 事件驱动 / 轮询 50-100ms | 部分可控 |
| HTTP 本地通信 | 亚毫秒 | 基本可控 |
| Python 处理 + SQLite | 无交易时 15ms | 可控 |
| MT5 IPC（positions_get 等） | 每次 ~1-10ms | 基本可控 |
| 经纪商 `order_send` | 100-500ms | **不可控** |

这培养了一个重要能力：**判断哪些问题在代码层面能解决，哪些问题必须换基础设施/外部环境**。440ms 的 `order_send` 再优化代码也压不下来——这是工程里最现实的判断。

---

## 3. 数据库写入优化的实战

为了把 15ms 压到 5ms 以下，用到了两个标准数据库优化手段：

- **WAL 模式**：`PRAGMA journal_mode = WAL;` 减少写盘同步阻塞
- **批量写入**：把多次 `commit()` 合并成一次 `executemany + commit`

这是计算机学习中很实用的内容，因为几乎所有系统都会碰到数据库写入瓶颈。

---

## 4. 缓存与去重思维

`CopyEngine` 里加的快照去重，本质上是一个**幂等性优化**：

> 同样的输入第二次来的时候，没必要重新计算。

这对应了分布式系统、接口设计、消息队列中常见的**幂等 + 去重**思想。

---

## 5. 测试驱动与最小改动

每次优化后都跑 `python -m unittest discover -s tests -v`，11 个测试全过。这是一个很好的工程习惯：

- 改一点，测一点
- 不因为优化破坏原有功能
- 用测试保护自己

---

## 6. 混合系统调试经验

一个真实系统里同时涉及：

- MQL4 脚本（EA）
- Python HTTP 服务
- MetaTrader5 的 C++/DLL IPC
- SQLite 本地存储
- 外部经纪商交易服务器

调试这种系统需要跨语言、跨进程的视角，这是单纯写算法题或 Web 后端很难获得的经验。

---

## 7. 工程权衡的真实案例

这次优化里有很多权衡：

- 降轮询到 50ms 会增加 CPU 但减少兜底延迟
- WAL 模式提升写入速度但增加 WAL 文件管理
- 快照去重跳过 sync，但重启后缓存丢失，要靠持久化 state 兜底

这些不是"对不对"的问题，而是"适不适合当前场景"的问题。

---

## 一句话总结

这次实战最值钱的地方不是把延迟从 15ms 压到 5ms，而是学会了**怎么定位瓶颈、怎么分层判断、怎么小步快跑地做优化而不破坏功能**——这套方法放到任何系统里都能用。
